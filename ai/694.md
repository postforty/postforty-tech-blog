# Hermes Agent와 gemini-3.1-flash-lite로 무장한 돌쇠(서브 에이전트) 군단으로 무료 에이전트 구축

이 글을 작성하는 2026년 6월 기준, `gemini-3.1-flash-lite` 모델의 무료 티어는 일 500회, 분당 5회로 호출 횟수가 엄격히 제한되어 있다. 이 모델을 메인 에이전트인 Hermes(애칭 '제이')에 연동하여 운영하다 보니 '429 에러'가 빈번하게 발생하여 시스템이 멈추는 일이 잦았다. 이를 근본적으로 해결하기 위해 제이에게 쿼터 제한을 우회할 '조수'를 만들도록 지시했다. Google AI Studio에서는 계정당 최대 10개의 프로젝트를 생성할 수 있으며, 각 프로젝트에서 발급된 API 키는 독립적인 1일 쿼터를 보장받는다. 즉, 10개의 키를 확보하면 하루 최대 5,000회까지 호출이 가능해진다. 이 점에 착안하여 제이에게 7개의 API 키를 제공하고 파이썬으로 7명의 조수(이하 '돌쇠')를 생성하도록 하였다. 이 돌쇠들은 특정 돌쇠의 쿼터가 소진되면 즉시 다음 돌쇠로 업무를 넘기는 '로테이션(Rotation) 시스템'을 성공적으로 구축했다. 다음은 제이가 구축하여 보고한 해당 아키텍처의 상세 내용이다.

이 시스템은 사장님의 전략적 의사결정을 지원하기 위해 '안정성, 투명성, 데이터의 팩트 기반 검증'을 최우선으로 설계되었습니다.

1. 시스템 핵심 구성 요소
- 엔진: Gemini 3.1 Flash-Lite (빠른 응답 및 저비용)
- 두뇌(Dolsoe Legion): gemini_assistant.py (독립적 API 키 로테이션 매니저)
- 검색(DDG): DuckDuckGo Search 도구 (무료/무제한 실시간 검색)
- 지휘 통제(Jay): 모든 데이터 수집 결과를 독립적으로 검증하고 사장님께 최종 요약 전달

2. 크리티컬 이슈 해결 및 아키텍처 개선
- 429 에러 완전 봉인: litellm 내부의 기본 재시도 루프를 차단하고, DolsoeManager에서 직접 제어권을 확보하여 에러 발생 시 즉시 다음 프로젝트 키로 순환(Round-Robin)하도록 로직을 재구성했습니다.
- 인증 매핑 정확도: 환경 변수 오염을 방지하기 위해 각 Dolsoe 객체가 생성 시점에 고유 API 키를 직접 소유하도록 하여, 구글 클라이언트의 키 주입 실패 문제를 해결했습니다.
- 환경 오염 제거: 시스템 업데이트나 환경 변화에 대응하기 위해 os.environ 주입 대신 '호출 시 API 키 강제 주입 방식'으로 아키텍처를 전면 교체했습니다.

3. 운용 프로토콜 (제이의 절대 철칙)
- 추측 금지: 데이터가 없거나 호출이 실패하면 "확인되지 않음"으로 즉시 보고합니다.
- 데이터 독립성: 검색된 뉴스(DDG)와 주가 수급 데이터를 강제로 엮지 않고, 유의미한 연결고리가 확인될 때만 매핑합니다.
- 신뢰성 입증: 돌쇠가 일을 수행했는지 구글 AI Studio 호출 이력을 기준으로 사장님께 매번 검증 결과를 알립니다.

5. 돌쇠 군단 운영 현황
