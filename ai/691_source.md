# 하네스 엔지니어링 심층 분석
## 1. AI 패러다임의 진화와 도입 배경
### 1-1. AI 활용 방법론의 3단계 변화
          + 1단계 프롬프트 엔지니어링
            모델에게 '어떻게 잘 물어볼 것인가'에 집중하던 시기로, 퓨샷(Few-shot), 페르소나 부여, 생각의 사슬(Chain of Thought) 등의 기법이 활용되었다. 이 단계에서는 AI가 조언자에 머물렀을 뿐 직접 행동하지는 못했다.
          + 2단계 컨텍스트 엔지니어링
            RAG(검색 증강 생성)나 외부 지식의 동적 결합을 통해 AI에게 필요한 정보를 적절히 깔아주는 데 집중한 시기이다. 컨텍스트 윈도우가 크게 확장되었으나, 장시간 자율적으로 작업을 수행하는 능력은 여전히 부족했다.
          + 3단계 하네스 엔지니어링
            AI가 말하던 시대에서 '일하는 시대'로 넘어가며 모델이 행동할 수 있는 주변 환경(Harness)을 설계하는 단계이다. 테라폼(Terraform)의 창시자인 하시모토 미첼(Mitchell Hashimoto)이 명시적으로 사용하며 정립된 이 개념은, 모델의 지능 자체보다 모델을 둘러싼 규칙과 도구 시스템을 개선하는 데 초점을 맞춘다.
#### 1-1-1. AI 에이전트의 4가지 주요 실패 패턴과 컨텍스트 부패
AI 모델에게 복잡한 애플리케이션 개발을 맡겼을 때 빈번하게 나타나는 실패 패턴은 모델의 추론 능력이 아닌 환경의 부재에서 기인한다.
          + 벼락치기(Cramming): 단계별 계획 없이 전체 작업을 한 번에 처리하려다 중간에 중단되는 현상이다.
          + 성급한 완료 선언: 완료 기준이 모호하여 작업이 끝나지 않았음에도 임의로 완료했다고 보고하는 패턴이다.
          + 검증 없는 마무리: 자가 점검 도구가 없어 본인이 만든 코드를 실행하거나 테스트하지 않은 채 결과물을 제출하는 현상이다.
          + 기억 상실: 이전 세션이나 전임자의 진행 기록 없이 백지상태로 작업을 시작하여 동일한 오류를 반복하는 현상이다.
          + 컨텍스트 부패(Context Rot)
            Chroma Research의 실험에 따르면, AI에게 주어지는 정보가 늘어날수록 특정 정보를 찾는 능력이 오히려 떨어지는 '컨텍스트 부패' 현상이 발생한다. 따라서 정보 과부하를 막고 환경을 깨끗하게 유지하는 시스템적 개입이 필수적이다.

## 2. 하네스(Harness)의 정의와 핵심 구성 요소
### 2-1. 하네스의 비유적 정의
하네스는 원래 경주마에게 씌우는 고삐, 안장, 말발굽 등의 주변 장비를 의미한다. AI 모델이 야생마나 훌륭한 엔진이라면, 하네스는 속도를 제어하는 고삐나 브레이크, 변속기 역할을 한다. LangChain의 Vivek Trivedy는 "에이전트에서 모델을 제외한 모든 것이 하네스(Agent = Model + Harness)"라고 정의했다.
### 2-2. 하네스의 5가지 핵심 장치
효과적인 하네스를 구성하기 위해서는 시스템 프롬프트, 도구, 미들웨어, 메모리, 서브 에이전트 등의 요소가 유기적으로 결합되어야 한다.
          + 인수인계 노트 (기억 보존)
            에이전트가 작업을 마칠 때마다 진행 상황을 요약한 아티팩트를 파일로 기록해 다음 세션이나 에이전트에게 전달하는 장치이다.
          + 안전한 실험실 (샌드박스)
            AI가 작성한 코드를 실제 프로덕션 시스템과 격리된 샌드박스 환경에서 실행하여 시스템 피해를 막는다.
          + 깔끔한 책상 (정보 과부하 방지)
            시작 시점에 필요한 도구(파이썬 설치 여부 등)와 디렉토리 구조만 명시적으로 주입하고, 불필요한 정보는 분리 보관하여 컨텍스트 부패를 막는다.
          + 업무 위임 (방화벽 역할)
            자료 조사나 단순 탐색 작업은 별도의 서브 에이전트에게 위임하여, 메인 에이전트가 처리해야 할 정보가 불필요하게 오염되는 것을 방지한다.
          + 자동 품질 검사 (실수 방지 훅)
            작업 완료를 선언하기 전에 체크리스트를 강제로 실행하여 실수를 탐지하고, 실패 시 다시 수정 루프로 돌려보내는 자율 루프 시스템이다.

## 3. 현장의 증명: 선도 기업들의 하네스 최적화 사례
### 3-1. LangChain의 벤치마크 도약
LangChain은 코딩 능력을 측정하는 Terminal Bench 2.0에서 모델(GPT-5.2-Codex)을 고정한 채 하네스만 수정하여 순위를 30위권(52.8%)에서 5위(66.5%)로 25계단이나 끌어올렸다.
          + 트레이스 분석 기반의 개선
            LangSmith의 'Trace Analyzer Skill'을 활용해 에이전트의 에러 패턴을 분석하고 하네스를 점진적으로 부스팅했다.
          + 자동 검증 미들웨어
            Ralph Wiggum 루프와 유사하게, 에이전트가 종료 의사를 표시할 때 개입하여 작업 요구 사항에 대한 검증을 강제하는 'PreCompletionChecklistMiddleware'를 삽입했다.
          + 파멸의 루프(Doom Loop) 방지
            파일당 편집 횟수를 추적하여 일정 횟수 이상 동일 파일을 수정하면 접근 방식을 전환하도록 경고하는 미들웨어를 도입했다.
            ```python
            # LoopDetectionMiddleware 개념적 예시
            def trace_file_edits(file_path, edits_count):
                if edits_count >= 10:
                    inject_context("같은 실수가 반복되고 있습니다. 접근 방식을 재검토하세요.")
                return continue_execution()
            ```
          + 리즈닝 샌드위치 (Reasoning Sandwich)
            계획 수립과 최종 검증 단계에는 추론 연산량을 최대로 높이고(xhigh), 단순 실행 단계에서는 중간 수준(high)으로 낮추는 토큰 최적화 전략을 사용했다.
### 3-2. Anthropic의 자율 코딩 및 디자인 평가 구조
Anthropic은 프론트엔드 디자인과 풀스택 코딩을 수행할 때 모델의 '컨텍스트 불안증'과 '자가 평가의 관대함'을 극복하기 위해 다중 에이전트 구조를 도입했다.
          + 생성과 평가 에이전트의 분리
            모델이 스스로의 결과물에 관대한 문제를 해결하기 위해 계획(Planner), 생성(Generator), 평가(Evaluator) 에이전트로 역할을 분리했다. 평가 에이전트는 Playwright MCP 같은 도구를 사용해 실제 UI와 API를 클릭하며 기능을 테스트한다.
          + 스프린트 계약 체결
            코드를 작성하기 전 생성 에이전트와 평가 에이전트가 작업 완료의 명확한 기준(Sprint contract)을 합의한 후, 한 번에 하나의 기능만 구현하도록 강제했다.
          + AI 슬롭(AI Slop)에 대한 명시적 페널티
            프론트엔드 디자인 시 흔한 AI 생성 패턴(예: 흰색 카드 위 보라색 그라데이션)을 사용하면 평가 에이전트가 명시적인 페널티를 주어 독창적이고 과감한 디자인을 유도한다.
### 3-3. 하시모토 미첼의 AI 도입 여정
하시모토 미첼은 단순 챗봇 사용에서 벗어나 에이전트를 상시 운영하는 6단계 여정을 거치며 하네스의 중요성을 강조했다.
          + 개인적 하네스 엔지니어링 구축
            에이전트가 잘못된 행동을 할 때마다 `AGENTS.md` 같은 지침 파일을 즉시 업데이트하거나 맞춤형 검증 스크립트를 제공하여 동일한 실수가 구조적으로 반복될 수 없도록 조치했다.
            ```bash
            # AGENTS.md 시스템 프롬프트 업데이트 예시
            1. 항상 테스트 코드를 먼저 작성할 것.
            2. 불필요한 파일 삭제 명령어(rm -rf) 사용 엄금.
            3. 특정 디렉토리 구조 밖으로 벗어나지 말 것.
            ```

## 4. 좋은 하네스를 평가하는 기준과 자동화
### 4-1. 하드 게이트와 소프트 페널티 시스템
좋은 하네스는 단순히 정답률이나 속도만으로 평가할 수 없으며, 안전성과 정책 준수 여부를 종합적으로 판단해야 한다.
          + 하드 게이트 (Hard Gate)
            평가 단계에서 하네스를 즉시 탈락시키는 절대적 안전 기준이다. 테스트 파일을 임의로 조작하거나, 보안상 위험한 명령어를 실행하거나, 정책상 불가능한 개인 정보 유출 및 환불 처리를 할 경우 성공 여부와 무관하게 즉시 탈락(Fail) 처리한다. 풀스택 평가 시 각 기준에 임계값(Hard threshold)을 두어 하나라도 미달하면 전체 스프린트를 실패 처리하는 방식도 이에 해당한다.
          + 소프트 페널티 (Soft Penalty)
            합격 후보들을 대상으로 최적화와 효율성을 채점하는 감점 기준이다. 기본 성공 보상에 토큰 사용량, 지연 시간, 불필요한 도구 호출, 할루시네이션 발생 횟수 등에 가중치를 곱해 감점한다.
### 4-2. 트레이스(Trace) 기반의 평가 자동화
하네스를 객관적으로 평가하고 개선하기 위해서는 에이전트의 모든 행동(명령어, API 호출, 읽은 파일 등)을 트레이스(흔적)로 기록해야 한다. 이 데이터를 바탕으로 실패 원인을 파악하고 메타-하네스(Meta-harness)가 자동으로 하네스 환경을 제안하고 평가하는 순환 구조를 만들 수 있다.

## 5. 클라우드 인프라 활용 및 미래 전망
### 5-1. AWS AgentCore와 AI-DLC
개별 팀이 하네스 시스템을 직접 구축하는 수고를 덜기 위해 클라우드 플랫폼 차원의 지원도 확대되고 있다.
          + AWS AgentCore 인프라
            Amazon Bedrock의 AgentCore는 에이전트의 메모리, 샌드박스(Code Interpreter), 정책 제어(Policy), 품질 자동 측정(Evaluations) 등 하네스의 핵심 요소를 9개의 인프라 서비스로 제공하여 진입 장벽을 낮추고 있다.
          + AI-DLC 방법론
            구상, 구축, 운영의 각 단계마다 AI가 세운 계획을 사람이 승인하도록 통제하는 소프트웨어 생명주기 관리 체계로, 철저한 검증 루프를 환경에 내재화한다.
### 5-2. 환경 설계자로의 진화
미래의 개발자는 단순히 코드를 치거나 프롬프트를 입력하는 것을 넘어, AI 에이전트가 어떤 도구를 쓰고 어떤 피드백 루프를 거칠지 전체 작업 환경을 설계하는 아키텍트(Architect)의 역할로 진화하고 있다. 결국 하네스 템플릿의 발전과 함께 "어떤 모델을 쓰느냐"만큼 "어떤 하네스를 구축하느냐"가 경쟁력을 가르는 핵심 지표가 될 것이다.

---

수집된 자료와 이전 대화를 바탕으로, 현장에서 AI 모델의 성능을 극대화하기 위해 구체적으로 하네스를 적용하는 방법들을 정리해 드립니다. 

하네스 적용의 핵심은 단순히 모델에게 지시를 내리는 프롬프트를 넘어서, 에이전트가 실수 없이 자율적으로 일할 수 있는 환경, 도구, 제어 시스템을 구축하는 것입니다. 구체적인 적용 방법은 다음과 같습니다.

**1. 컨텍스트와 정보의 엄격한 통제 (깔끔한 책상 및 기억 보존)**
*   **시작 시점의 명시적 환경 주입 (로컬 컨텍스트 미들웨어):** 에이전트가 파일 탐색 등에 시간을 낭비하거나 엉뚱한 가정을 하지 않도록, 작업 시작 시 디렉토리 구조, 파이썬이나 패키지 매니저 같은 사용 가능한 도구, 그리고 제한된 시간 예산(Time budget) 등의 환경 정보를 명시적으로 먼저 주입합니다.
*   **컨텍스트 초기화 및 인수인계 노트 활용:** 긴 작업 시 모델이 이전 지시를 잊는 '컨텍스트 부패'나 빨리 작업을 마무리하려는 '컨텍스트 불안증'을 막기 위해, 세션 간에 대화 기록을 넘기지 않고 **진행 상황과 다음 단계를 요약한 '아티팩트(인수인계 노트)'만 파일로 넘겨주는 방식**을 사용합니다.

**2. 에이전트 역할 분리와 '스프린트 계약' 도입**
*   **계획-생성-평가의 다중 에이전트 구조:** 모델이 자신의 코드를 관대하게 평가하는 문제를 해결하기 위해 3개의 에이전트(Planner, Generator, Evaluator)로 역할을 분리합니다.
*   **스프린트 계약 (Sprint Contract):** 코드를 작성하기 전에 생성 에이전트와 평가 에이전트가 한 번에 하나의 기능(스프린트) 단위로 **무엇이 '완료' 상태인지 명확한 검증 기준을 먼저 합의**합니다. 이후 평가 에이전트가 Playwright 같은 자동화 도구를 사용해 실제 UI나 API를 클릭해 가며 합의된 기준을 엄격하게 테스트합니다.

**3. 미들웨어(Middleware)와 훅(Hook)을 통한 자동 개입**
*   **종료 전 강제 자가 검증 (Pre-Completion Checklist):** 에이전트가 임의로 작업을 끝냈다고 선언하는 것을 막기 위해, 종료 시점을 가로채는 훅을 달아 테스트 코드 실행과 엣지 케이스 점검 등 필수 체크리스트를 강제로 수행하게 만듭니다(일명 Ralph Wiggum Loop).
*   **파멸의 루프(Doom Loop) 방지:** 에이전트가 잘못된 코드에 갇혀 같은 파일을 미세하게 계속 수정하는 현상을 막기 위해, 파일당 수정 횟수를 추적하여 일정 횟수(예: 10회)를 넘기면 **"접근 방식을 재검토하라"는 경고 컨텍스트를 주입하는 미들웨어**를 적용합니다.

**4. 추론 자원의 전략적 분배 (추론 샌드위치 기법)**
모든 작업에 최대의 컴퓨팅 자원을 쓰면 타임아웃이나 비용 낭비가 발생합니다. 따라서 작업을 세분화하여, 깊은 사고가 필요한 **'초기 계획(Planning)'과 '최종 검증(Verification)' 단계에는 추론 자원을 최대로 할당(xhigh)**하고, 실제 코드를 짜는 **'실행(Build)' 단계에는 중간 수준(high)으로 낮추는 샌드위치 방식(xhigh-high-xhigh)을 적용**해 성능과 효율을 최적화합니다.

**5. 시스템적 피드백 구축 및 `AGENTS.md` 업데이트**
*   **맞춤형 도구 제공 및 규칙 강제:** 에이전트가 자주 하는 실수가 발견되면 부탁하는 프롬프트를 길게 쓰는 대신, 아예 실수를 검증할 수 있는 필터링 테스트 스크립트나 스크린샷 캡처 도구를 하네스에 추가합니다.
*   **간단한 약도 형태의 지침 파일:** 수백 장의 매뉴얼 대신 100줄 내외의 명확한 `AGENTS.md` 파일을 프로젝트 내에 두어, 에이전트가 "불필요한 파일 삭제 금지", "테스트 코드 필수 작성" 같은 핵심 규칙을 즉각 참조할 수 있게 구조화합니다.

이처럼 하네스는 계획, 생성, 테스트, 에러 수정으로 이어지는 루프(Plan-Build-Verify-Fix)를 자동화하고, 에이전트가 한계를 벗어나지 않도록 시스템 차원에서 강제하는 모든 방법을 포괄합니다.
